## Part 3: Platform Architecture

### 3.1 Overview

The *Through Two Sets of Eyes* platform has three components that need to work together without depending on any single person, account, or paid service to survive:

**GitHub** - kit repository, version control, issue tracking, contributor wiki. Oriented toward facilitators, contributors, and kit developers.

**WordPress** - public-facing website, project gallery, facilitator forum, project map, site registration form, data submission forms. Oriented toward participants, the public, potential facilitators, and research partners.

**Google Workspace** - data collection backend. Google Forms receive SWEMWBS submissions, image submissions, site completion forms, and site registration forms. Google Sheets aggregate and store the data. Oriented toward the project coordinator.

These three components are deliberately decoupled. A problem with any one of them does not take down the others. The GitHub repository remains accessible if the WordPress site goes down. The Google Sheets data remains intact if the GitHub repository is unavailable. No single failure point kills the project.

**Key files:**

*README.md* - the project's front door. Plain language, no jargon. Covers what the project is, how to run it, how to contribute, and where to get help. Should be readable by someone who has never heard of Photovoice or GitHub.

*LICENSE* - MIT licence, full text.

*CONTRIBUTING.md* - how to raise an issue, propose a change, submit a translation, or contribute an adaptation. Includes the core design principles that cannot be changed without forking the project.

*CHANGELOG.md* - a human-readable record of what changed between versions and why. Important for facilitators who have run the project before and want to know what's different in the current release.

*infrastructure/google-forms/form-definitions/* - XLSForm-compatible definitions of all Google Forms used in the project, stored as CSV. This means the forms can be reconstructed from scratch if the Google account is lost or migrated. Each form definition includes field names, field types, validation rules, and the hidden pre-fill parameters used in QR code generation.

*infrastructure/wordpress/* - documentation of the WordPress installation including required plugins, theme, custom fields, and page structure. Not the WordPress installation itself - that lives on the hosting server - but enough documentation that the site could be rebuilt from scratch by a competent WordPress user.

*infrastructure/backup/* - documented backup procedures and restoration instructions. See Section 3.5.

**Issue tracking**

GitHub Issues is the primary support and feedback channel for facilitators, contributors, and the public. Issue templates are provided for:

- Bug report - something in the kit is wrong or unclear
- Facilitation question - a situation the guide doesn't cover
- Translation contribution - offering a new language pack
- Adaptation contribution - offering a cultural or contextual adaptation
- Feature request - a suggested improvement to the kit or platform
- Protocol deviation report - documenting a significant deviation from the standard protocol at a completed site

Issue templates prompt the reporter to provide enough context that the project coordinator can respond usefully. All issues are public by default - a facilitator's question becomes searchable by future facilitators with the same question.

The GitHub Wiki serves as the knowledge base for settled questions - FAQ, known jurisdiction-specific adaptations, lessons learned from completed sites. Issues that generate useful resolved answers are summarised in the Wiki by the project coordinator.

**Releases and versioning**

The kit uses semantic versioning - MAJOR.MINOR.PATCH:

- MAJOR version increments for changes to core design principles or the SWEMWBS administration protocol - facilitators who have run a previous major version should re-read the Facilitator Guide before running again
- MINOR version increments for new components, new appendices, or significant expansions to existing sections
- PATCH version increments for corrections, clarifications, and minor improvements

Each release is tagged in GitHub and announced via the project mailing list. The current release is always linked prominently from the README and the WordPress site.

**Governance**

The GitHub organisation has a minimum of two owners at all times. If an owner becomes inactive the remaining owner is responsible for recruiting a replacement before their own involvement ends. This requirement is documented in the CONTRIBUTING.md file and reviewed annually.

Proposed changes to core design principles require a documented rationale and a minimum two-week comment period via a GitHub Issue before being merged. All other changes follow standard pull request review.

### 3.3 WordPress Site

**Hosting:** Self-hosted WordPress on a shared hosting account - Hostmonster/Bluehost or equivalent. WordPress is chosen for longevity, wide community support, and low technical barrier to handover.

**Domain:** `thru2setsofeyes.org` or equivalent - to be confirmed at launch.

**PHP and MySQL requirements:** Standard WordPress requirements - any shared hosting plan from a major provider will meet them.

---

**Required Plugins**

*Contact Form 7 (already installed)* - the primary form plugin for the WordPress site. Free. Handles site registration, site status updates, and general contact. CF7 submissions are emailed to the project coordinator and optionally stored in the WordPress database via the CF7 Database Plugin add-on. For forms that need to push data to Google Sheets, a Zapier or Make.com webhook can be configured - both have free tiers adequate for this project's volume.

*User Submitted Posts (already installed)* - handles front-end content submission from facilitators. Used for the community contributions section where facilitators share lessons learned, cultural adaptation notes, and site completion stories. All submissions are moderated by the project coordinator before publication. Replaces BuddyPress for V1.0 - a full forum may be considered for V2.0 if facilitator numbers warrant it.

*FooGallery* - lightweight image gallery plugin for the public exhibition gallery. Free tier available. Must be configured to support custom fields per image (site, week, prompt, cohort, caption) and filtering by those fields. Mobile-responsive display and lazy loading are required - both are supported in FooGallery's free tier.

*Leaflet Maps Marker* - displays the project site map showing registered and completed sites worldwide. Uses OpenStreetMap tiles, avoiding Google Maps API key management and usage costs. Free.

*WP Super Cache* - page caching plugin. Required from launch to ensure the image gallery loads acceptably on shared hosting. Free.

*UpdraftPlus* - automated backup plugin. Configured to send daily database backups and weekly full backups to Backblaze B2 remote storage. Free tier is adequate. See Section 3.5.

*Wordfence* - security plugin providing firewall, malware scanning, and login protection. Free tier is adequate for this project.

*WP Mail SMTP* - ensures WordPress notification emails are delivered reliably. Configured to use Brevo (formerly Sendinblue) as the transactional email provider. Both the plugin and Brevo's free tier are adequate for this project's email volume.

*WP Data Access* - displays aggregated SWEMWBS statistics on the public data dashboard, connected to the master Google Sheet via the Google Sheets API. Free tier available.

#### The Public Gallery

The gallery is the most important public-facing element of the WordPress site. It should feel like an exhibition, not a database.

Default view: a grid of paired images - one youth, one older adult - from the most recently completed week across all active sites. Visitor can filter by site, week, or prompt.

Each image displays:
- The image itself
- The prompt it responds to
- The participant's caption
- The site name and week number
- The cohort - youth or older adult - indicated by a subtle visual marker, not a label

No participant names appear in the gallery. The authorship reveal is a live event at each site's Week 8 closing - it is not replicated in the online gallery.

Image uploads to the gallery are managed by the project coordinator in V1.0, reviewed from the Google Drive folder fed by the participant image submission Google Form. Upload cadence is weekly, aligned with the print preparation cycle.

*Performance note:* Image galleries are the most common cause of slow WordPress sites. All images should be resized to a maximum of 1200px on the longest edge before upload. The gallery plugin should be configured for lazy loading. A caching plugin - WP Super Cache is free and adequate - should be installed from launch.

#### The Project Map

An interactive map showing every registered site as a pin, colour-coded by status:

- Blue: registered, not yet started
- Green: in progress
- Gold: completed, data submitted
- Grey: incomplete

Clicking a pin shows: site name (if permission given), location type, cohort sizes, project dates, and a link to that site's gallery images.

The map is updated automatically when facilitators submit site status updates via the Contact Form 7 Forms status form.

---

#### The Public Data Dashboard

A simple page showing aggregated statistics across all completed sites:

- Total sites completed
- Total participants
- Mean SWEMWBS scores at baseline, mid-point, and post-project - by cohort and overall
- Distribution of site types - metropolitan, regional, rural, remote
- Countries represented

Displayed as simple summary statistics and basic charts - not raw data. Updated monthly by the project coordinator from the master Google Sheet.

A plain-language interpretation note sits above the statistics: "These figures combine data from all completed sites. Individual site data is not shown separately. The project is at an early stage and these figures will become more meaningful as more sites contribute data."

### 3.4 Google Workspace

Google Workspace provides the data collection backend. It is chosen for V1.0 because it is free, widely accessible, works well on mobile, and requires no server administration.

**The dependency risk is acknowledged.** Google could change its terms, pricing, or products at any time. The mitigation is documentation - all form definitions are stored in the GitHub repository in a format that could be reconstructed in an alternative platform. The master Google Sheet is exported to CSV monthly and stored in the backup location described in Section 3.5.

---

#### Forms and Sheets

Five Google Forms, each feeding a tab in a single master Google Sheet:

*Form 1 - SWEMWBS Data Entry*
Completed by facilitators after each survey round. Fields as described in Section 2.4 Document 4. Auto-calculates metric scores in the Sheet via a formula column. One submission per participant per timepoint.

*Form 2 - Participant Image Submission*
Completed by participants directly via weekly QR code links. Fields as described in Section 2.6. Images stored in a linked Google Drive folder, organised automatically by site code and week number via a Google Apps Script triggered on form submission.

*Form 3 - Site Registration*
Completed by prospective facilitators via the WordPress site registration page. Fields as described in Section 2.8. Triggers an email notification to the project coordinator on submission.

*Form 4 - Site Completion*
Completed by facilitators after Week 8. Fields as described in Section 2.7. Includes a file upload field for session notes.

*Form 5 - Facilitator Session Notes*
A lightweight weekly form completed by facilitators after each session. Five fields: site code, week number, session date, attendance count, and a free text notes field. Designed to take under three minutes on a phone. Replaces the paper session notes template for facilitators who prefer digital entry - paper template remains available as an alternative.

---

#### Google Apps Script Automations

Three small automations handle the tasks that would otherwise require manual intervention after each form submission:

*Image organisation script* - triggered on Form 2 submission. Moves the submitted image from the default Google Drive upload folder into a structured folder hierarchy: `submissions / [site-code] / week-[n] /`. Renames the file to `[site-code]-w[n]-[participant-code].[ext]`. This makes the weekly curatorial review straightforward - the project coordinator opens the week's folder and sees all submissions organised by site.

*Site registration notification script* - triggered on Form 3 submission. Sends a formatted email to the project coordinator summarising the new registration and including a one-click link to approve and generate site codes.

*Monthly Sheet export script* - runs on a monthly schedule. Exports the SWEMWBS tab of the master Sheet as a CSV file and saves it to a designated backup folder in Google Drive. The backup folder is shared with at least one other project owner independently of the primary Google account.

*Submission window management script* - triggered on a weekly schedule aligned to each site's session day. Sets the current week's image submission Google Form to active at the close of the session window and inactive 48 hours before the next session. In V1.0 this script runs on a fixed weekly schedule - sites that run sessions on different days will need to coordinate their submission windows with the project coordinator at registration. Automating per-site scheduling is a V2.0 priority.

*Site completion confirmation script* - triggered on Form 4 (Site Completion) submission. Sends a formatted confirmation email to the facilitator's registered email address acknowledging receipt of their submission and listing the data fields received. Also sends a notification email to the project coordinator. This replaces the need for the facilitator to manually verify that their submission was received.

---

#### Access Control

The master Google Sheet and all linked Google Drive folders are owned by the project coordinator Google account and shared with at least one other project owner with full edit access. No other parties have access to the raw data.

Form submission confirmations are visible to respondents - a plain-language confirmation screen and optional email receipt. No raw data is visible to respondents.

The image submission Google Drive folder structure is not publicly shared. Images move from there to the WordPress gallery only via the project coordinator's review process.

### 3.5 Backup and Longevity

This section is the most important in Part 3. The project is designed to outlast any individual's involvement. That requires documented, tested, automated backup of everything that matters.

---

#### What Needs to Be Backed Up

| Asset | Location | Backup method | Frequency |
|---|---|---|---|
| Kit documents | GitHub | Git history - inherently version controlled | Continuous |
| WordPress database | Hosting server | UpdraftPlus to remote storage | Daily |
| WordPress files | Hosting server | UpdraftPlus to remote storage | Weekly |
| Master Google Sheet | Google Drive | Apps Script CSV export | Monthly |
| Submitted images | Google Drive | Google Takeout export | Monthly |
| Form definitions | GitHub repository | Manual update on form change | On change |
| WordPress documentation | GitHub repository | Manual update on site change | On change |

---

#### Remote Storage for WordPress Backups

UpdraftPlus should be configured to send backups to a remote storage location independent of the hosting account. Recommended options in order of preference:

*Backblaze B2* - very low cost (approximately USD $0.006 per GB per month), S3-compatible, UpdraftPlus supports it natively, not linked to the hosting account.

*Google Drive* - free up to 15GB, UpdraftPlus supports it natively, but introduces a dependency on the same Google account used for data collection. Acceptable as a secondary backup destination, not recommended as the primary.

*A second hosting account* - if a project co-owner has their own hosting, a copy of the WordPress backup there provides strong independence. Belt and braces.

#### Handover Documentation

The GitHub repository contains a `HANDOVER.md` file - a plain-language document written for someone taking over the project coordinator role with no prior involvement. It covers:

- What the project is and what the coordinator does week to week
- Access credentials location - not the credentials themselves, but instructions for where they are stored and how to transfer them
- The monthly maintenance checklist
- How to generate QR code card sets for new sites
- How to perform the weekly image curation and gallery upload
- How to update the public data dashboard
- Who to contact for hosting, Google Workspace, and GitHub questions
- How to transfer GitHub organisation ownership
- How to transfer the WordPress hosting account

The handover document is reviewed and updated annually by the current project coordinator. It is the single most important document for project longevity and should be treated accordingly.

---

#### The Minimum Viable Continuity Test

Once per year the project coordinator should confirm that:

- The WordPress site is accessible and all plugins are updated
- The UpdraftPlus backup has run successfully in the past seven days
- The monthly Google Sheet export is running and files are accumulating in the backup folder
- At least one other person has access to the backup storage location independently of the primary account
- The HANDOVER.md file reflects the current state of the project
- The GitHub organisation has at least two active owners

This is a 30-minute annual checklist, not an audit. It is noted in the GitHub repository as an annual task with a due date.
