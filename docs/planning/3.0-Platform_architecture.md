# Through Two Sets of Eyes - Project Specification

## Part 3: Platform Architecture

### 3.1 Overview

The *Through Two Sets of Eyes* platform has three components that need to work together without depending on any single person, account, or paid service to survive:

**GitHub** - kit repository, version control, issue tracking, contributor wiki. Oriented toward facilitators, contributors, and kit developers.

**WordPress** - public-facing website, project gallery, facilitator forum, project map, site registration form. Oriented toward participants, the public, potential facilitators, and research partners.

**Supabase** - PostgreSQL database, file storage, REST API, and coordinator authentication. Oriented toward the project coordinator and the data collection pipeline.

These three components are deliberately decoupled. A problem with any one of them does not take down the others. The GitHub repository remains accessible if the WordPress site goes down. The Supabase data remains intact if the GitHub repository is unavailable. No single failure point kills the project.

**Issue tracking**

GitHub Issues is the primary support and feedback channel. Issue templates are provided for:

- Bug report - something in the kit is wrong or unclear
- Facilitation question - a situation the guide doesn't cover
- Translation contribution - offering a new language pack
- Adaptation contribution - offering a cultural or contextual adaptation
- Feature request - a suggested improvement to the kit or platform
- Protocol deviation report - documenting a significant deviation from the standard protocol

All issues are public by default. The GitHub Wiki serves as the knowledge base for settled questions.

**Releases and versioning**

Semantic versioning - MAJOR.MINOR.PATCH:

- MAJOR - changes to core design principles or SWEMWBS administration protocol
- MINOR - new components, new appendices, significant expansions
- PATCH - corrections, clarifications, minor improvements

**Governance**

Minimum two GitHub organisation owners at all times. Proposed changes to core design principles require a two-week comment period via GitHub Issue before merging.

#### Required Plugins

*Contact Form 7 (already installed)* - handles site registration form, site status update form, and general contact. Submissions are emailed to the project coordinator. CF7 is used for forms where the data does not need to go directly into Supabase - the coordinator processes registrations manually and enters the site code into Supabase after review.

*User Submitted Posts (already installed)* - handles the facilitator community contributions section. Facilitators submit lessons learned, adaptation notes, and site completion stories. All submissions moderated before publication.

*FooGallery* - lightweight image gallery plugin. Displays approved images fetched from the Supabase `gallery` storage bucket. Configured for lazy loading and mobile-responsive display. Free tier.

*Leaflet Maps Marker* - interactive project site map using OpenStreetMap tiles. Site pins populated from the Supabase `public_site_map` view via JavaScript fetch. Free.

*WP Super Cache* - page caching. Required from launch to ensure acceptable performance on shared hosting. Free.

*UpdraftPlus* - automated WordPress database and file backups to Backblaze B2 remote storage. Note: this backs up the WordPress installation only - Supabase data is backed up separately. Free tier.

*Wordfence* - security plugin. Free tier.

*WP Mail SMTP* - reliable email delivery via Brevo. Free tier.

#### The Public Gallery

Images in the gallery are served from the Supabase `gallery` storage bucket - a public bucket containing only coordinator-approved images. The gallery page fetches image metadata from the `public_gallery` view via the Supabase JavaScript client, then constructs image URLs from the bucket.

Default view: a grid of paired images - one youth, one older adult - from the most recently published week across all active sites. Filterable by site, week, and prompt.

Each image displays:
- The image itself - served directly from the `gallery` bucket URL
- The prompt it responds to
- The participant's caption
- The site name and week number
- The cohort indicated by a subtle visual marker

No participant names appear in the gallery.

*Performance note:* All images are resized to a maximum of 1200px on the longest edge before being copied to the gallery bucket. The FooGallery plugin is configured for lazy loading. WP Super Cache handles page-level caching.

---

#### The Project Map

An interactive Leaflet map populated from the Supabase `public_site_map` view via JavaScript fetch on page load. Site pins are colour-coded by status:

- Blue: registered, not yet started
- Green: in progress - week number shown in popup
- Gold: completed, data submitted
- Grey: incomplete

The `public_site_map` view automatically respects the `venue_name_public_ok` flag - venue names only appear in popups where the facilitator has given permission.

---

#### The Public Data Dashboard

A simple page showing aggregated statistics fetched from the Supabase `public_dashboard` view:

- Total sites completed
- Total participants
- Mean SWEMWBS metric scores at baseline and post-project, by cohort
- Distribution of site types
- Countries represented

The JavaScript fetch runs on page load. WP Super Cache should be configured to exclude this page from caching, or set to a short cache lifetime, so statistics update promptly as new sites complete.

A plain-language interpretation note sits above the statistics noting that figures combine all completed sites and will become more meaningful as more sites contribute data.

### 3.4 Supabase

Supabase provides the PostgreSQL database, file storage, REST API, and coordinator authentication for the project. It is chosen because:

- Fully managed PostgreSQL 15 - actively maintained, SSL everywhere by default
- REST API auto-generated from the schema - no custom backend code needed for standard operations
- Row Level Security enforced at the database level - not dependent on application code
- Supabase Storage for image files - two buckets with separate access policies
- Open source - the entire platform can be self-hosted if needed
- Free tier is sufficient for V1.0 and likely V2.0

**Project URL:** `https://mybqgaimdeluvgjegmmj.supabase.co`
**Credentials:** Supabase anon key and service key stored in Bitwarden. See HANDOVER.md Section 5.

---

#### Database Schema

Six tables and four views. The complete schema is in `infrastructure/supabase/schema.sql` in the kit repository. Running this file on a fresh Supabase project recreates the entire database.

**Tables:**

| Table | Purpose | Key constraint |
|---|---|---|
| `sites` | Master site registry | `site_code` unique |
| `swemwbs_surveys` | SWEMWBS responses | One per participant per timepoint |
| `image_submissions` | Participant image metadata | One per participant per week |
| `session_notes` | Facilitator weekly notes | One per site per week |
| `site_completions` | Post-project data | One per site |
| `swemwbs_conversion` | Raw-to-metric lookup | 29 rows, populated once |

**Views:**

| View | Purpose | Access |
|---|---|---|
| `public_dashboard` | Aggregate SWEMWBS statistics | Public |
| `public_site_map` | Site registry for map display | Public |
| `public_gallery` | Approved published images | Public |
| `moderation_queue` | Pending image submissions | Authenticated only |

---

#### Row Level Security

RLS is enabled on all six tables. The policy summary:

| Table | Anon insert | Anon select | Authenticated |
|---|---|---|---|
| `sites` | Yes | No | Full access |
| `swemwbs_surveys` | Yes | No | Full access |
| `image_submissions` | Yes | No | Full access |
| `session_notes` | Yes | No | Full access |
| `site_completions` | Yes | No | Full access |
| `swemwbs_conversion` | No | Yes | Full access |

Anon insert is enabled to allow form submissions without requiring facilitators or participants to create accounts. The application layer performs additional validation - site code existence, participant code format, week number range - before inserting.

The `status_update_key` in the `sites` table is a 64-character random hex string generated on insert. Facilitators use this key to update their site status via a dedicated form without requiring authentication.

---

#### Storage Buckets

Two buckets with separate access policies:

**`submissions` bucket - private**

Participant images land here on upload. Not publicly accessible. The coordinator reviews submissions via the `moderation_queue` view and the Supabase dashboard.

Policies:
- Anon: insert only
- Authenticated: select, insert, delete

File path convention: `[site_code]/week-[n]/[site_code]-w[n]-[participant_code].[ext]`

**`gallery` bucket - public**

Approved images are copied here by the coordinator after moderation. Publicly accessible - images are served directly from bucket URLs embedded in the WordPress gallery page.

Policies:
- Anon: select only
- Authenticated: select, insert, delete

---

#### Data Flows

**Site registration:**
CF7 form on WordPress > email to coordinator > coordinator reviews > coordinator inserts to `sites` table via Supabase dashboard > coordinator emails site code, facilitator code, and status update key to facilitator

*Note for V2.0:* Automate the insert and email via a Supabase Edge Function triggered by a webhook from CF7.

**SWEMWBS data entry:**
Facilitator opens web form on phone > form validates site code and facilitator code against `sites` table > form calculates raw total, looks up metric score from `swemwbs_conversion` > inserts to `swemwbs_surveys`

**Participant image submission:**
Participant scans QR code > opens image submission form in browser > form uploads image directly to `submissions` bucket > inserts metadata row to `image_submissions` with `moderation_status = 'pending'`

**Submission window management:** 
The image submission form page checks the 
current date against the site's session schedule before accepting submissions. 
The coordinator sets each site's session day at registration. The form page 
queries the `sites` table for the session day and calculates whether the 
current submission window is open - from session end until 48 hours before 
the next session. Submissions outside the window are accepted but flagged 
in `image_submissions` with a note that they may miss the print deadline. 
In V1.0 the coordinator reviews flagged submissions and contacts the 
facilitator if a reprint is needed.

**Image moderation:**
Coordinator reviews `moderation_queue` view > opens image from `submissions` bucket > approves: resizes image, copies to `gallery` bucket, updates `moderation_status = 'approved'` and `gallery_published = true` in `image_submissions` > rejects: updates `moderation_status = 'rejected'`, optionally deletes from `submissions` bucket

**Session notes:**
Facilitator opens session notes form on phone > inserts to `session_notes`

**Site completion:**
Facilitator opens site completion form > inserts to `site_completions` > Supabase sends confirmation email via SMTP settings

**Site status update:**
Facilitator uses status update link from registration email > form validates `status_update_key` against `sites` table > updates `status` and `current_week` fields

---

#### Coordinator Authentication

The Supabase dashboard provides full database and storage access for the coordinator using the project account credentials. No separate coordinator application is needed for V1.0.

For moderation tasks the coordinator:
1. Signs into the Supabase dashboard
2. Uses the `moderation_queue` view to identify pending submissions
3. Opens images from the `submissions` bucket in the Storage browser
4. Approves or rejects via SQL update or the Table Editor
5. Copies approved images to the `gallery` bucket

A lightweight coordinator web interface - a password-protected HTML page using the Supabase service key - is planned for V2.0 to make moderation accessible without Supabase dashboard access.

### 3.5 Backup and Longevity

The project is designed to outlast any individual's involvement. That requires documented, tested backup of everything that matters.

---

#### What Needs to Be Backed Up

| Asset | Location | Backup method | Frequency |
|---|---|---|---|
| Kit documents | GitHub | Git history - inherently version controlled | Continuous |
| WordPress database | Hosting server | UpdraftPlus to Backblaze B2 | Daily |
| WordPress files | Hosting server | UpdraftPlus to Backblaze B2 | Weekly |
| Supabase database | Supabase | Automatic - 7 day retention on free tier | Daily (automatic) |
| Supabase database | Coordinator held | `pg_dump` via direct connection string | Monthly |
| Supabase storage | Supabase | Not automatically backed up on free tier | Monthly manual export |
| Schema and policies | GitHub repository | `infrastructure/supabase/` directory | On change |

---

#### Supabase Database Backup

Supabase free tier includes automatic daily backups with 7-day retention. These are accessible via Supabase > Settings > Backups and can be restored by Supabase support.

For coordinator-held backups independent of Supabase, run monthly via SSH or a local terminal:

```bash
pg_dump \
  "postgresql://postgres:[PASSWORD]@db.mybqgaimdeluvgjegmmj.supabase.co:5432/postgres" \
  --no-owner \
  --no-acl \
  -f thru2setsofeyes-backup-$(date +%Y-%m-%d).sql
```

Store the resulting SQL file in Backblaze B2 alongside the WordPress backups. This file can restore the entire database to any PostgreSQL 15 instance.

---

#### Supabase Storage Backup

Images in the `submissions` and `gallery` buckets are not automatically backed up on the free tier. Monthly export procedure:

1. Supabase dashboard > Storage > `gallery` bucket
2. Select all files > Download
3. Store the downloaded archive in Backblaze B2

The `submissions` bucket contains only unmoderated images - rejected or unreviewed. These are less critical than the gallery but worth exporting quarterly.

#### Schema and Policies as Code

The complete database schema and all RLS and storage policies are stored in the GitHub repository:

- `infrastructure/supabase/schema.sql` - all CREATE TABLE, CREATE VIEW, INSERT statements
- `infrastructure/supabase/policies.sql` - all RLS policies and storage bucket policies

Running these two files in sequence on a fresh Supabase project recreates the entire database infrastructure. This is the disaster recovery procedure if the Supabase project is lost or corrupted.

Update these files whenever the schema or policies change. They are the infrastructure source of truth.

---

#### Migrating Away from Supabase

If Supabase becomes unavailable or changes its pricing, migration procedure:

1. Export the database via `pg_dump` as above
2. Provision a new PostgreSQL 15 instance - self-hosted via Docker, or another managed provider
3. Restore via `psql [new-connection-string] < thru2setsofeyes-backup-[date].sql`
4. Update the project URL and anon key in all form pages and WordPress JavaScript
5. Update HANDOVER.md with new connection details

The Supabase client library is a thin wrapper around standard PostgreSQL and REST - switching providers requires updating connection strings and keys, not rewriting application logic.
